Research Studio Query Rewrite Benchmark
Date: 2026-05-08

1. Implementation Summary

This change adds a structured Query Rewrite layer before local RAG retrieval. It does not replace the user query used for routing, memory, persistence, or answer synthesis. The original query is retained as the first retrieval query, and rewrite variants are used only as additional weighted retrieval queries.

Implemented strategies:
- baseline_expansion: keeps the previous planned_search_queries behavior.
- standalone: resolves short or pronoun-heavy queries into a standalone retrieval query.
- step_back: generates a higher-level abstract query for principles, architecture, tradeoffs, or taxonomy.
- lexical_domain_expansion: combines tokenized sparse keywords with domain terms.
- hyde: generates a HyDE-style hypothetical answer/document query for dense retrieval.

Drift controls:
- Query variants have weights. Original/baseline queries keep weight 1.0; abstract and HyDE queries are lower-weight recall boosters.
- Lookup/taxonomy queries do not use HyDE, so author/title/category questions remain precise.
- R-DeepSC semantic-noise taxonomy and robustness queries are separated to avoid pulling in unrelated DeepSC low-SNR chunks.
- Live tools, memory writes, TODO creation, and calculator routing continue to use the raw user query.

Main changed paths:
- backend/app/models.py: QueryRewriteVariant and QueryRewriteResponse schemas.
- backend/app/services.py: rewrite generation, weighted retrieval merge, Plan-and-Solve/Agent/LATS local RAG integration.
- backend/app/config.py, .env.example, docker-compose.yml, README.md, README.zh-CN.md: runtime config.
- backend/tests/test_query_rewrite.py: rewrite regression tests and semantic-communication benchmark.


2. Benchmark Setup

Corpus:
- 8 synthetic semantic-communication assets.
- 20 chunks total: 17 gold chunks and 3 distractor chunks covering:
  - semantic communication vs Shannon communication
  - DeepSC text semantic communication
  - R-DeepSC semantic noise and robustness
  - DeepSC-ST speech tasks
  - task-oriented semantic communication and edge inference
  - DIB/DDIB and selective retransmission
  - high-speed semantic speech transmission, superimposed pilot, SwitchAC-SIP
  - future 6G semantic-native architecture, multimodal trends, security/privacy
  - distractors: Shannon capacity, Semantic Web, ordinary ASR

Question set:
- 15 main semantic-communication questions.
- 6 paraphrase questions for stability.
- Topics include architecture, technology choices, robustness, metrics, compression, high-speed channels, future trends, and distractor suppression.

Metrics:
- hit@1/hit@3/hit@5: whether any gold chunk appears in top-k.
- MRR@5: reciprocal rank of the first gold chunk.
- coverage@5: fraction of required gold chunks covered by top-5.
- avg_rank: average first gold rank.
- rank_stdev: lower means retrieval rank is more stable.
- drift_rate: top-1 landing on a distractor chunk.

Key comparison against baseline:
- hit@1 improved from 0.800 to 0.933.
- MRR@5 improved from 0.869 to 0.967.
- coverage@5 improved from 0.967 to 1.000.
- average first gold rank improved from 1.467 to 1.067.
- rank stdev improved from 1.087 to 0.249.
- drift_rate dropped from 0.133 to 0.067.
- hit@5 stayed at 1.000, so rewrite did not lose top-5 recall.
- The 3-query hybrid setting produced the same main metrics as hybrid_full, so the headline gain is not dependent on simply using more queries than baseline.

6. Limitations

- This benchmark supports a narrow claim: Query Rewrite improves retrieval stability on the current synthetic semantic-communication benchmark.
- It does not prove broad cross-domain superiority. The corpus is small, synthetic, and focused on one technical domain.
- The benchmark and rewrite rules are in the same repository, so there is some overfitting risk.
- Most main questions were already strong under baseline; the largest main-set gains come from Q13 and Q14. The strongest overall evidence is the paraphrase set, where rewrite improves coverage and rank stability.
- Future validation should run the same metrics on real uploaded papers/notes and additional domains such as codebase QA, project planning, and general literature review.
